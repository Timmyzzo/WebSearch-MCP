# WebSearch MCP 开发路线与验收标准

## 1. 项目定位

本项目目标是构建一个可被 Cherry Studio、Claude Code、Codex 等主流 MCP 客户端稳定调用的通用网络搜索 MCP。

核心能力收敛为：

- Grok：负责理解问题、执行联网搜索并组织答案。
- Tavily Search：补充结构化搜索结果与可靠信源。
- Tavily Extract：抓取网页正文并输出 Markdown。
- Tavily Map：发现网站结构与页面 URL。

项目优先级依次为：可用性、答案准确性与前沿性、故障可解释性、跨客户端兼容性、性能与维护性。

当前进度：P0、P1、P2、P3、P4 已完成；下一阶段为 P5 搜索 Prompt 与搜索质量重构。

## 2. 已确认的产品决策

- [x] 完整移除旧抓取服务，包括 Search、Scrape、配置项、降级逻辑、文档和遗留工具描述。
- [x] 不再实现旧抓取服务自动降级，网页抓取仅使用 Tavily Extract。
- [x] Tavily 多 Key 采用轮询作为正常调度方式。
- [x] Tavily Key 出错时按状态码和响应内容分类，采用不同熔断策略。
- [x] Grok 支持主模型和备用模型；主模型连续失败后自动切换备用模型。
- [x] MCP 必须兼容 Cherry Studio、Claude Code 和 Codex，核心能力不得依赖某个客户端的私有配置。
- [ ] 当前阶段不调整敏感日志策略，先保证功能稳定和搜索质量。
- [x] 后续代码提交目标仓库为 `https://github.com/Timmyzzo/WebSearch-MCP`。

## 3. 跨客户端兼容要求

### 3.1 通用协议要求

- [x] 以标准 MCP stdio transport 作为基础运行方式。
- [x] 工具参数使用稳定、简单的 JSON Schema，避免客户端特有类型。
- [x] 工具正常结果和错误结果都返回结构化对象，避免让客户端解析混合文本。
- [x] `Context` 等客户端上下文必须是可选能力；没有 Context 时工具仍可正常运行。
- [x] 不要求调用方暴露完整思维过程，不把 `thought` 作为必填参数。
- [x] 不在 MCP 核心工具中修改 Claude Code、Codex 或 Cherry Studio 的本地配置。
- [x] 移除客户端专属配置工具；此类客户端配置改为文档说明。

### 3.2 兼容性验收矩阵

- [ ] Cherry Studio：能发现工具、传递参数、显示结构化结果和错误。
- [ ] Claude Code：能通过 stdio 启动、搜索、抓取、映射和读取信源。
- [ ] Codex：能通过 stdio 启动，长时间调用不会因父进程监控或信号处理异常退出。
- [ ] 三个客户端对同一工具的必填参数、默认值和返回字段理解一致。
- [x] 为三个客户端分别提供可复制的安装与环境变量配置示例。

## 4. Tavily 多 Key 调度与熔断

### 4.1 正常调度

- [x] `TAVILY_API_KEYS` 支持逗号、分号和换行分隔。
- [x] 正常请求按 Key 轮询，避免单个 Key 长期承担全部流量。
- [x] 日志和错误消息只显示 Key 指纹，例如前后各 4 位，不输出完整 Key。
- [x] Key 状态在 Search、Extract、Map 三类调用之间共享。

### 4.2 错误分类

| 类型 | 典型信号 | 策略 |
| --- | --- | --- |
| Key 永久失效 | 401、403，或响应明确表示 Key 无效/被撤销 | 立即熔断该 Key，本进程内不再使用；提醒用户重新生成；继续尝试下一个 Key |
| 单 Key 限流或额度耗尽 | 429，且响应表明配额、额度或 Key 级限流 | 将该 Key 放入冷却或额度耗尽状态；立即尝试下一个 Key |
| Tavily 临时服务异常 | 408、500、502、503、504、连接失败、超时 | 不判定 Key 永久失效；短暂退避重试，并使用其他 Key 验证是否为服务级故障 |
| 请求参数错误 | 400、422 | 不轮换所有 Key；直接返回参数错误，避免无效消耗全部 Key |
| API 地址或路由错误 | 404，或所有 Key 对同一端点得到一致错误 | 判定为配置/版本问题，停止当前调用并提示检查 `TAVILY_API_URL` |

状态码只能作为第一层信号。实现时还应解析 Tavily 的错误码、响应正文和 `Retry-After`，避免把服务整体故障误判为 Key 永久失效。

### 4.3 熔断状态

每个 Key 至少维护以下状态：

- `healthy`：可正常轮询。
- `cooldown`：临时限流或暂时异常，到期后允许半开探测。
- `quota_exhausted`：当前额度周期内不可用，可配置较长冷却时间。
- `invalid`：Key 已撤销或认证失败，本进程内永久停用。

服务整体还应维护 Tavily 全局熔断状态：

- 当多个不同 Key 都出现相同 5xx 或网络错误时，优先判断为 Tavily 服务异常。
- 全局熔断期间快速失败，避免每次调用都等待完整超时。
- 冷却期后允许一次半开探测，成功则恢复服务。

### 4.4 所有 Key 不可用时

- [x] 立即停止当前 MCP 工具调用，不终止整个 MCP 服务进程。
- [x] 返回稳定错误码，例如 `tavily_all_keys_unavailable`。
- [x] 返回各 Key 的脱敏状态摘要，指出哪些无效、哪些限流、哪些暂时不可用。
- [x] 明确提醒用户补充或重新生成 Tavily Key。
- [x] `web_fetch` 和 `web_map` 直接失败；`web_search` 保留 Grok 部分结果时以 P2 最小扩展 `partial=true` 和 `tavily_error` 明确标记。

## 5. Grok 主备模型与重试

### 5.1 配置目标

建议增加：

- `GROK_PRIMARY_MODEL`：能力强、作为默认搜索模型。
- `GROK_FALLBACK_MODEL`：主模型不可用时使用的备用模型。
- `GROK_MODEL_MAX_ATTEMPTS`：每个模型最多实际调用次数，建议默认 3。
- 保留 `GROK_MODEL` 作为兼容配置，迁移期内映射到主模型。

“最多 3 次”统一定义为每个模型最多 3 次实际调用，主模型与备用模型分别计数，最坏情况下共 6 次调用。该值应允许通过环境变量调整。

### 5.2 调用状态机

1. 使用主模型调用。
2. 对可恢复错误执行带随机抖动的指数退避，主模型最多调用 3 次。
3. 主模型仍失败时，记录脱敏故障原因并切换备用模型。
4. 备用模型独立执行最多 3 次调用。
5. 备用模型仍失败时停止当前工具调用，返回结构化错误，让调用 MCP 的主模型向用户解释。

### 5.3 Grok 错误分类

- [x] 408、429、5xx、连接错误、读取超时、流中断：允许重试。
- [x] 中转站返回可识别的“上游账号不可用/死号”错误：允许重试，以便中转站重新路由。
- [x] 模型不存在、模型暂时不可用：可提前切换备用模型。
- [x] 明确的请求参数错误：不重复相同请求，直接报告。
- [x] 明确的 API Key 认证失败：通常不应切换模型反复请求，因为主备模型共用同一认证；应直接提示检查 Grok API Key。若中转站使用特殊错误语义，应允许通过规则配置。

### 5.4 最终错误输出

建议格式：

```json
{
  "status": "error",
  "error": {
    "code": "grok_primary_and_fallback_failed",
    "message": "主模型和备用模型均不可用",
    "primary_model": "...",
    "fallback_model": "...",
    "attempts": 6,
    "last_error_type": "upstream_unavailable"
  }
}
```

不得伪装成空搜索结果，也不得返回看似成功但 `content` 为空的响应。

## 6. 搜索质量目标

### 6.1 总体原则

搜索结果应追求：

1. 前沿性：优先当前版本、近期研究和最新一手资料。
2. 准确性：关键结论必须有可追溯证据，区分事实、推断和经验观点。
3. 科学性：优先高等级证据，主动寻找反例、限制条件和不同结论。
4. 全面性：覆盖主要观点，但不以大量低质量来源制造虚假全面。
5. 专业性：优先真正从事该领域的机构、研究者、工程团队和专业从业者。

### 6.2 通用信源等级

优先顺序如下：

1. 官方文档、标准、法规、原始数据、论文原文、系统综述和 Meta 分析。
2. 权威机构、大学、实验室、专业协会、项目维护者和一线专业团队。
3. 有编辑审核和事实核查机制的专业媒体。
4. 专业人士的公开经验分享，作为实践补充并明确其证据等级。
5. 普通博客、论坛和社交媒体只用于发现线索，不单独支撑关键结论。

来源数量不能替代来源质量。多个转载同一原始消息的页面只能视为一个证据来源。

### 6.3 软件开发与 GitHub 问题

- [ ] 优先搜索项目当前默认分支的官方文档、README、Release、Changelog 和迁移指南。
- [ ] 核对最新版本号、发布日期、相关 commit、issue 和 pull request。
- [ ] 区分稳定版、预览版和已废弃接口。
- [ ] 回答中注明适用版本和检索日期，避免用旧文档解释新版本。
- [ ] 第三方教程只能补充示例，不应覆盖项目官方行为。

### 6.4 健身、运动表现与饮食

- [ ] 科学依据优先使用系统综述、Meta 分析、随机对照研究以及 ACSM、NSCA 等专业组织资料。
- [ ] 顶尖扣篮选手、NBA 训练师、职业队体能教练的方案作为高价值实践经验补充。
- [ ] 明确区分“研究支持”“专家实践”“运动员个人经验”，不能把冠军经验直接等同于普适因果证据。
- [ ] 结合训练年龄、伤病史、体重、力量基础、恢复和设备条件给出适用范围。
- [ ] 涉及伤病、疾病、药物或极端饮食时提示专业医疗评估边界。

以“提高弹跳”为例，应同时覆盖力量、爆发力、反应力量、技术、周期安排、疲劳管理和伤病风险，而不是只罗列网红动作。

### 6.5 汽车安全等高风险生活问题

- [ ] 优先使用 IIHS、NHTSA、Euro NCAP、ANCAP 等碰撞测试与官方事故数据。
- [ ] 结合真实道路死亡率、车型年份、车重、车身类别、主动安全配置和事故类型。
- [ ] 不直接横向比较不同测试年份或不同标准下的星级。
- [ ] 区分碰撞避免能力与碰撞发生后的乘员保护能力。
- [ ] 明确统计限制和不确定性，不承诺“绝对安全”。

### 6.6 小众与突发奇想问题

- [ ] 先拆解概念和判定标准，再进行多语言、多角度搜索。
- [ ] 主动搜索反例、失败案例和不同学派观点。
- [ ] 至少用两类相互独立的来源交叉验证关键结论。
- [ ] 如果高质量资料稀缺，应明确告诉用户证据不足，而不是用普通博客填满答案。

## 7. 搜索规划与 Prompt 策略

- [ ] 搜索深度应随问题复杂度和风险动态调整，不强制所有问题执行同样数量的搜索。
- [ ] 简单事实和单一文档问题走快速路径。
- [ ] 高风险、强时效、专业或小众问题启用广度搜索、深挖和反证搜索。
- [ ] 涉及“最新、当前、今天、现版本”时注入当前日期并检查资料更新时间。
- [ ] 不要求每一句话都附引用，但所有关键事实、数字、医学/训练建议和安全结论必须能映射到来源。
- [ ] 最终答案标注证据等级、争议点、适用条件和已知限制。
- [ ] 规划能力应尽量内部化或合并成一个可选工具，避免六阶段规划工具占用大量 MCP 调用。

## 8. 缓存与内容大小策略

缓存的目标是降低延迟，不能牺牲前沿性。

- [ ] 不长期缓存最终搜索答案。
- [ ] 模型列表可短期缓存，并设置 TTL；临时失败不能永久缓存为空结果。
- [ ] Tavily Key 健康状态和冷却时间只缓存运行状态，不持久化完整 Key。
- [ ] 搜索信源会话增加 TTL 和最大数量，避免长期运行后无限占用内存。
- [ ] 规划会话增加 TTL 和最大数量。
- [ ] 网页内容过大时采用结构化截断或分块，不允许无标识地截断。
- [ ] 保留 URL、标题、抓取时间、是否截断和剩余内容提示。

初期实现以合理的固定默认值为主，不为缓存和分块引入过度复杂的分布式组件。

## 9. 建议工具集合

核心工具建议控制为：

- `web_search`：Grok 主搜索，可选 Tavily 补充信源。
- `get_sources`：读取某次搜索的完整信源。
- `web_fetch`：仅使用 Tavily Extract 抓取内容。
- `web_map`：使用 Tavily Map 发现站点结构。
- `get_config_info`：返回脱敏配置、模型和服务健康状态。
- `switch_model`：可选；需要明确它修改的是主模型还是备用模型。
- `plan_search`：可选的统一规划工具，替代多个阶段工具。

客户端专属设置工具不进入核心工具集合。

P4 的兼容实现约定：

- `status` 是权威状态；规范错误对象位于 `error_detail`。
- `error_detail` 至少包含 `code`、`message`、`service` 和 `retryable`，存在时还包含 `http_status`、`upstream_code` 和脱敏 `diagnostics`。
- 旧 `error` 字符串、`partial`、`content`、`results`、`success`、P2 `tavily_error` 和 P3 `grok_error` 暂不删除。
- Grok 成功、Tavily 补充失败为 `partial_success`；Grok 最终失败始终为 `error`，Tavily 结果不能替代 Grok 答案。
- Grok 有效答案没有来源仍为 `success`；空答案、空抓取内容和空 URL 映射分别使用稳定错误码，不作为空成功。
- 结构化错误只终止当前调用；标准 stdio 初始化、工具发现、后续调用和可选 `Context` 保持可用。

## 11. 暂缓事项

以下事项当前不阻塞可用版本，后续再评估：

## 12. 完成定义

第一阶段可用版本应满足：

- Cherry Studio、Claude Code、Codex 均可稳定调用。
- 旧抓取服务无任何运行时代码和配置残留。
- Tavily 正常轮询，多种错误能正确分类并切换 Key。
- 所有 Tavily Key 不可用时，当前调用快速停止并明确提醒用户。
- Grok 主模型失败后自动切换备用模型，双模型失败时返回清晰错误。
- 所有核心与保留规划工具使用统一三态和通用脱敏错误对象，并保留 P2/P3 兼容字段。
- 空内容、空 URL 和异常失败不会伪装成成功；工具错误后 MCP 进程仍可继续服务。
- 搜索结果优先最新、权威和原始资料，并能解释证据等级与限制。
- 自动化测试覆盖关键故障路径。
